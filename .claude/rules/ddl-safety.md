# DDL Safety Guarantee

## The Safety Rules

**All generated PostgreSQL DDL MUST follow these rules without exception:**

### 1. Always Quote Identifiers

Every table name, column name, index name, and constraint name MUST be wrapped in double quotes (`"identifier"`). This:
- Prevents SQL injection via crafted names
- Handles PostgreSQL reserved words (`user`, `order`, `group`, `table`, `column`, etc.)
- Preserves mixed-case names exactly as defined

```sql
-- CORRECT
CREATE TABLE "users" (
  "id" BIGINT GENERATED ALWAYS AS IDENTITY,
  "order" VARCHAR(100),  -- reserved word, safe because quoted
  "group_id" INTEGER NOT NULL
);

-- NEVER DO THIS
CREATE TABLE users (
  id BIGINT,
  order VARCHAR(100)  -- BREAKS: "order" is a reserved word
);
```

### 2. Validate Identifier Names

Before generating DDL, validate all identifiers:
- Max 63 characters (PostgreSQL limit)
- No null bytes
- Must start with letter or underscore (after quoting, PG allows more, but enforce this for sanity)
- Warn on reserved words (still quote them, but show a warning in the UI)
- Reject names containing `"` (double quote) â€” cannot be safely quoted
- Reject names containing `;`, `--`, `/*` â€” injection patterns

### 3. Transaction Wrapping

All DDL exports MUST be wrapped in a transaction:

```sql
-- Generated by DB Diagram Visualizer
-- Date: 2024-01-15T10:30:00Z
SET client_encoding = 'UTF8';

BEGIN;

-- Enum types
CREATE TYPE "user_status" AS ENUM ('active', 'inactive', 'banned');

-- Tables (ordered by FK dependencies)
CREATE TABLE "roles" ( ... );
CREATE TABLE "users" ( ... );  -- references "roles"

-- Indexes
CREATE INDEX "users_email_idx" ON "users" ("email");

COMMIT;
```

### 4. FK Dependency Ordering (Topological Sort)

Tables MUST be created in dependency order:
1. Tables with no FK references first
2. Tables whose FK targets are already created next
3. If circular FKs exist: create tables WITHOUT FK constraints first, then `ALTER TABLE ADD CONSTRAINT` in a second pass

```sql
-- Circular FK handling
BEGIN;

-- Pass 1: Create tables without circular FKs
CREATE TABLE "employees" (
  "id" BIGINT GENERATED ALWAYS AS IDENTITY,
  "name" TEXT NOT NULL,
  "manager_id" BIGINT  -- FK added in pass 2
);

-- Pass 2: Add circular FK constraints
ALTER TABLE "employees"
  ADD CONSTRAINT "employees_manager_id_fkey"
  FOREIGN KEY ("manager_id") REFERENCES "employees" ("id");

COMMIT;
```

### 5. CREATE TYPE Before CREATE TABLE

Enum types and composite types MUST be created before any table that references them:

```sql
CREATE TYPE "status" AS ENUM ('active', 'inactive');
-- THEN
CREATE TABLE "users" (
  "status" "status" NOT NULL DEFAULT 'active'
);
```

### 6. CHECK Expression Safety

CHECK constraint expressions are the highest injection risk. Rules:
- Only allow: column references, comparison operators (`=`, `<>`, `<`, `>`, `<=`, `>=`), logical operators (`AND`, `OR`, `NOT`), `IS NULL`, `IS NOT NULL`, `IN (...)`, `BETWEEN`, numeric/string literals, `LIKE`, `SIMILAR TO`
- REJECT any expression containing: `;`, `--`, `/*`, `*/`, `DROP`, `ALTER`, `INSERT`, `UPDATE`, `DELETE`, `EXECUTE`, `COPY`, `\\`
- Validate parentheses are balanced
- Maximum expression length: 500 characters

### 7. DEFAULT Value Safety

DEFAULT values must be validated against the column type:
- Numeric types: only numeric literals, `0`, `NULL`
- String types: only string literals (single-quoted), `NULL`
- Boolean: only `TRUE`, `FALSE`, `NULL`
- UUID: only `gen_random_uuid()`, `NULL`
- Timestamp: only `NOW()`, `CURRENT_TIMESTAMP`, `NULL`, ISO date literals
- REJECT any DEFAULT containing: `;`, `--`, `/*`, subqueries, function calls not in the whitelist

### 8. IF NOT EXISTS â€” Optional, Off by Default

`IF NOT EXISTS` is a convenience guard. Include it only when the user explicitly enables it. Never include `DROP TABLE` or `DROP TYPE` in generated DDL unless the user explicitly requests it with a confirmation.

## Anti-Patterns (NEVER DO)

- Never concatenate user input directly into SQL strings
- Never generate `DROP` statements without explicit user confirmation
- Never skip identifier quoting for "simple" names â€” always quote everything
- Never trust CHECK or DEFAULT expressions without validation
- Never generate DDL with unresolved FK targets (referencing tables that don't exist in the schema)
- Never output partial transactions (BEGIN without COMMIT)
